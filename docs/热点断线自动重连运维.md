# Jetson 热点断线自动重连运维

本文只处理 Jetson 与已保存 Wi-Fi 热点的断线恢复，不启动 ROS bringup、Nav2、巡逻，不发布
`/cmd_vel`，也不重启 NetworkManager。真实断热点验收必须由现场人员在小车静止、无导航、
无巡逻时执行。

## 1. 故障表现与根因

热点关闭或短暂消失后，Jetson 失去 Wi-Fi；热点恢复时，NetworkManager profile 虽可能已
启用 `autoconnect`，但默认有限重试、低优先级和无线节能会使重新关联不可靠。最终 Jetson
长期没有重新获得热点 IPv4，APP、SSH 和网页调试均无法恢复。

当前现场 profile `xxfq1` 的只读检查结果为：`autoconnect=yes`、
`autoconnect-retries=-1`（使用 NetworkManager 默认值）、`autoconnect-priority=0`、
`powersave=default`。本方案把持久连接属性修正为无限重试、明确优先级和禁用 Wi-Fi 节能，
再用 systemd timer 每隔 10 秒做一次安全兜底。

## 2. 设计原理与安全边界

恢复分两层：

1. NetworkManager 原生 `connection.autoconnect` 是主恢复机制。
2. `ylhb-wifi-reconnect.timer` 每轮只执行一次检查。仅当目标接口空闲、目标 SSID 可见时，
   才执行一次 `nmcli --wait 20 connection up <profile> ifname <interface>`。

watchdog 不做无限循环，不用公网 ping，不因热点没有互联网而重连，不断开有线或其他
Wi-Fi，不切换到其他热点，不修改 route metric、IP、网关、DNS、防火墙、SSID 或密码，
不重启 NetworkManager，也不关闭/开启 Wi-Fi radio。目标连接已 active 但还没有 IPv4 时，
只等待 NetworkManager DHCP。

## 3. 确认 profile 名称和无线接口

profile 名称不是 SSID。不要根据 SSID 猜 profile 名称。

```bash
nmcli -f NAME,UUID,TYPE,DEVICE,AUTOCONNECT connection show
nmcli -t -f NAME,TYPE,DEVICE connection show --active
```

选择顺序为：`YLHB_WIFI_CONNECTION`、当前活动 Wi-Fi、唯一保存的 Wi-Fi。存在多个候选时，
脚本拒绝自动选择，必须显式设置变量。

无线接口不要假设为 `wlan0`：

```bash
nmcli -t -f DEVICE,TYPE device | awk -F: '$2=="wifi"{print $1}'
```

当前 Jetson 检测到的接口是 `wlP1p1s0`。

## 4. status、dry-run、apply 和 restore

```bash
cd /home/nvidia/ros2_DL
export YLHB_WIFI_CONNECTION='实际连接profile名称'
export YLHB_WIFI_INTERFACE='实际无线接口'

./scripts/configure_wifi_reconnect.sh status
./scripts/configure_wifi_reconnect.sh dry-run
sudo -E ./scripts/configure_wifi_reconnect.sh apply
```

默认设置：

```text
connection.autoconnect=yes
connection.autoconnect-retries=0
connection.autoconnect-priority=100
802-11-wireless.powersave=2
```

可用变量：

```bash
export YLHB_WIFI_AUTOCONNECT_PRIORITY=100
export YLHB_WIFI_RECONNECT_INTERVAL_SEC=10
```

脚本默认保留 `connection.permissions`。若 status 显示 `user:nvidia:`，systemd root 服务可能
无法使用该 profile；只有现场明确决定改为系统级连接时才执行：

```bash
export YLHB_WIFI_MAKE_SYSTEM_CONNECTION=true
sudo -E ./scripts/configure_wifi_reconnect.sh apply
```

原值在修改前备份到 `/home/nvidia/.config/ylhb/wifi_reconnect_backup.json`，目录权限 `0700`，
文件权限 `0600`。恢复按 UUID 定位，即使 profile 后续改名也不会恢复错连接：

```bash
sudo -E ./scripts/configure_wifi_reconnect.sh restore
```

apply/restore 都只写 profile 属性，不会自动 down/up 当前连接。由现场人员决定是否重新激活。

## 5. 安装、状态与日志

```bash
sudo -E ./scripts/install_wifi_reconnect_service.sh
systemctl status ylhb-wifi-reconnect.timer --no-pager
journalctl -u ylhb-wifi-reconnect.service -n 50 --no-pager
```

安装脚本把运行副本安装到 `/usr/local/libexec/ylhb-wifi-reconnect`，避免 root 服务直接执行
仓库内用户可写脚本。非敏感配置写入 `/etc/ylhb/wifi-reconnect.env`，权限 `0600`，只包含
profile 名称和接口，不包含密码。timer 安装后启用，安装过程不手动执行 `connection up`。

连续看不到 SSID 属于正常等待，不记录 error；同类扫描、配置或激活错误至少 30 秒才记录
一次。连接失败后由 timer 下一轮重试。

## 6. 热点改名或密码改变

SSID 或密码改变后，旧 profile 不再匹配。由现场人员通过 Jetson 图形界面或 `nmtui` 更新/
新建正确的 NetworkManager profile，不要把密码写入仓库、日志或服务 env。随后重新确认
profile 名称；若名称或接口变化，重新导出变量并运行安装脚本以更新
`/etc/ylhb/wifi-reconnect.env`。

脚本不会删除旧 profile，也不会自动切换到名称相似的热点。

## 7. 安全回退

仅卸载 watchdog，保留 env 和 NetworkManager 备份：

```bash
sudo ./scripts/uninstall_wifi_reconnect_service.sh
```

同时删除 `/etc/ylhb/wifi-reconnect.env`：

```bash
sudo ./scripts/uninstall_wifi_reconnect_service.sh --purge-config
```

恢复 NetworkManager profile 原值需单独执行：

```bash
sudo -E ./scripts/configure_wifi_reconnect.sh restore
```

卸载不会删除用户连接，不恢复/修改有线和 5G 配置，也不删除
`/home/nvidia/.config/ylhb/wifi_reconnect_backup.json`。

## 8. 现场断线验收

先确认小车静止、无导航、无巡逻，再由现场人员执行：

1. 保持 Jetson 开机。
2. 关闭电脑或手机热点。
3. 等待 30 秒。
4. 重新开启同名热点。
5. 不操作 Jetson。
6. 验证 Jetson 在热点恢复后 30 秒内自动连接并重新获得 IPv4。
7. 验证 SSH、APP WebSocket 和网页调试恢复。
8. 验证机器人 ROS 节点未被重启。
9. 验证有线/5G 连接未受影响。

通过标准：无需本地操作 Jetson；不重启 NetworkManager；不切断其他网络；不启动任何
机器人运动；APP 和 SSH 可重新连接。
