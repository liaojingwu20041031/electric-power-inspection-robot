# 低成本测试规范

本仓库是学生实训性质的 ROS2 电力巡检机器人项目。测试的作用是保护安全红线和稳定契约，而不是追求数量、覆盖率或企业级流程。

当前自研代码中已经存在约 40 余个测试文件/测试脚本，测试密度明显高于项目当前的实机成熟度。后续默认进入“测试冻结”状态：除安全关键、核心协议或已复现回归外，不再新增测试文件。

## 总体判断

ROS2 机器人既需要自动化测试，也必须承认自动化测试的边界。

自动化测试适合验证确定性逻辑，例如：解析、格式、状态转换、命令路由、几何计算和配置契约。它无法证明真实机器人在场地中能够正确转弯、避障、定位、制动或识别目标。

因此本项目采用三层验证：

1. 快速构建与静态检查。
2. 少量、定向的自动化测试。
3. 仿真、rosbag 和真实机器人验收。

## 风险分类

| 级别 | 场景 | 默认做法 |
|---|---|---|
| A 安全关键 | 急停、控制锁、轮速方向、危险恢复动作、速度上限 | 必须保留自动化测试，并进行实机验收 |
| B 核心契约 | 路线 schema、地图语义、巡逻状态机、移动端/API 字段 | 优先修改已有测试，最多补少量用例 |
| C 普通确定性逻辑 | 解析器、路径处理、纯函数、文件转换 | 有真实回归时才补测试 |
| D UI 与普通配置 | 文案、布局、按钮顺序、DWB scale、等待时间 | 不新增测试 |
| E 硬件与动态行为 | 电机、传感器、TF 时序、Nav2 动态、SLAM、视觉效果 | 实机、仿真或 rosbag，不用复杂 mock 替代 |

## 应保留的自动化测试

- 急停、控制锁、命令白名单和危险命令拒绝。
- 左右轮速度与方向映射、底盘几何、footprint。
- NMEA 等纯协议解析器。
- 地图灰度到 free/occupied/unknown 的转换语义。
- 巡逻启动、暂停、恢复、取消、失败和终态重启。
- 路线与地图绑定、schema/API 字段兼容性。
- 地图保存、预览生成、确定性坐标或几何算法。

## 不应继续扩张的测试

- 完整中文句子、帮助文本、日志措辞。
- QML 控件顺序、固定高度、列数、折叠区域的具体结构。
- 普通 Nav2/SLAM 参数的精确小数值。
- 单纯检查某个源文件是否出现字符串的重复静态测试。
- 每新增一个小类就创建一个新的测试文件。
- 为测试搭建比业务代码还复杂的 fake、fixture 或 mock 框架。
- 测试第三方 `zed-ros2-wrapper`、雷达驱动等 vendor 内部实现。

## 当前仓库审查结论

### `ylhb_base`

总体价值最高，应保留但减少精确调参断言。

建议保留：

- `test_zlac_drive_mapping.cpp`
- `test_wtrtk980_nmea.cpp`
- `test_robot_geometry.py`
- `test_nav2_localization_config.py` 中的 footprint、地图 unknown、scan topic、速度安全边界和危险恢复行为
- 必要的 IMU/RTK 启动参数契约

建议放松：

- inflation、采样数、critic scale、等待时间、普通 resolution 等可调值的精确相等断言。
- 只要参数存在、类型正确、范围合理且跨组件一致即可。

### `ylhb_llm`

测试碎片化最明显。Agent、语音、UI、启动脚本被拆成大量文件，后续禁止继续按类创建新测试文件。

建议将新增行为归入已有边界：

- 工具和命令：`test_agent_tools.py`
- 策略与危险动作：`test_agent_policy.py`
- 决策协议：`test_agent_schema.py`
- Agent 端到端纯逻辑：`test_inspection_agent_runtime.py`
- ROS 节点去重、错误发布：`test_inspection_agent_node.py`
- UI backend：`test_ui_backend.py`

`test_qml_navigation.py` 只保留关键 backend 绑定、命令路由和危险旧路径不存在。中文文案、视觉结构、帮助文本和组件细节不应成为稳定契约。

`test_run_on_jetson.py` 只需要保护正式启动入口和关键安全开关。帮助信息每一行、模式名称列表和示例文本不需要全部锁死。

### `ylhb_mobile_bridge`

路线存储、地图管理、巡逻执行器和 schema 具有实际价值。优先保留状态机、数据持久化和 API 字段；进程管理和状态文本只测公开结果，不测内部调用顺序。

### `ylhb_3d_mapping`

纯文件管理、SVO 参数生成和确定性元数据可以测试；ZED 真机录制质量、重建质量、点云完整性和 Jetson 性能必须实测。

## 新增测试硬门槛

新增自动化测试必须同时满足：

1. 能明确描述所防止的真实事故或回归。
2. 被测行为无硬件也能稳定复现。
3. 现有测试没有覆盖同一契约。
4. 测试不依赖网络、在线模型或大型数据。
5. 可以定向执行，通常在一分钟内完成。
6. 测试维护成本低于一次人工验收。

不满足任一项，就改写为实机/仿真验收步骤。

## 数量预算

- 普通修改：0 个新测试文件。
- 单个 bug 修复：优先在已有文件增加 1 个回归用例。
- 新增确定性核心模块：最多 1 个新测试文件，通常 1～3 个核心用例。
- 跨包安全功能：可以增加必要测试，但必须在提交说明中列出风险场景。
- 不允许因为 Codex 建议“补充覆盖率”就自动扩张测试。

## 最快开发流程

### 日常修改

```bash
colcon build --packages-select <package> --cmake-args -DBUILD_TESTING=OFF
```

Python 逻辑修改只跑对应测试：

```bash
python3 -m pytest -q path/to/test_file.py::test_name
```

### 功能完成

```bash
colcon test --packages-select <package>
colcon test-result --verbose
```

### 阶段交付

跨包重构、里程碑或演示前再运行全仓库测试，并完成实机清单。

## 实机验收清单模板

每次只选择与修改相关的项目：

- 节点能启动，无持续 ERROR/FATAL。
- 关键 topic、service、action 名称正确。
- TF 树连续，时间戳没有明显跳变。
- 架空或低速确认左右轮方向和前后方向。
- 急停能立即阻止运动命令。
- 传感器断开、重连和权限异常表现可接受。
- Nav2 能完成目标点、转弯、避障、暂停和取消。
- 地图与巡逻点在真实场地中对齐。
- 视觉检测记录漏检、误检和推理帧率。
- UI/语音命令最终触发的真实动作正确。

## 清理策略

不要专门发起一次“大删测试”重构，这会消耗额度并可能引入新问题。采用随改随清：

1. 修改某模块时先检查同目录测试是否重复。
2. 能合并到已有测试文件就不新增文件。
3. 删除明显只测文案、布局或旧实现的断言。
4. 保留能够复现历史真实故障的测试。
5. 每次清理只覆盖当前功能范围，不做无关扩张。

## Agent 自检

- 是否先说明了测试保护的风险？
- 是否默认选择了“不新增测试”？
- 是否只运行受影响测试，而不是全量扫描？
- 是否把硬件动态问题留给实机、仿真或 rosbag？
- 是否避免了精确文案、布局和普通调参数值断言？
- 是否没有为了测试而过度重构业务代码？
