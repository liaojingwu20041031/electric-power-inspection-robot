# 好测试规范

测试保护真实故障，不追求数量。这个仓库优先保住会伤车、伤地图、伤巡逻流程、伤用户判断的行为；普通 UI 文案、布局细节、调参小数值不要锁死。

## 该写测试的地方

- 安全关键：急停、控制锁、速度/方向映射、底盘几何、footprint、禁止危险恢复动作。
- Nav2/SLAM 配置：地图 unknown 语义、`allow_unknown`、scan topic、local/global footprint 一致、速度上限和平滑器一致、参数名契约。
- 巡逻状态机：启动/暂停/继续/取消命令路由、循环等待、执行器不可用、终态可重启、不会因读属性发布命令。
- API/schema：移动端和 UI 依赖的字段名、类型、关键默认值。
- 地图保存/预览：205 灰度 unknown、预览图存在性/有效性、QML 不把图片 URL 当正文展示。
- UI 测试：测语义绑定和命令映射，不测完整中文句子、列数、高度、组件顺序。

## 少写或不写的地方

- 纯展示文案，除非它是 QML/API 契约。
- 精确布局：`columns`、固定高度、控件顺序。
- 普通调参：DWB critic scale、采样数、等待时间，只需要存在、为正、或在安全范围。
- 为 fake/mock 搭复杂框架。能用一个小 fake 类就不要引入 fixture 层。

## 分层

- 配置测试：读 YAML/XML/地图文件，验证安全红线和兼容性。
- 单元测试：验证纯状态转换、字段格式化、命令发布。
- QML 静态测试：验证页面入口、backend 属性绑定、命令字符串和危险旧路径不存在。
- 集成/硬件测试：默认不跑实机；需要硬件时单独说明前置条件。

## Codex 写测试决策流程

1. 先问：这个测试会挡住哪一种真实事故或回归？
2. 已有测试能覆盖就改已有测试，不新建同义测试。
3. 先测公开行为，再测内部实现。
4. 文案和布局只在它们是协议时测试。
5. 配置值优先测范围和一致性，只有安全阈值才测精确值。
6. fake 保持短小，只实现被测路径实际调用的方法。

## 自检清单

- 失败信息能指向真实问题。
- 改普通文案、布局或小幅调参不会导致无意义失败。
- 安全红线仍然被锁住。
- 没有为了测试改业务代码。
- 没有新增默认硬件依赖。
